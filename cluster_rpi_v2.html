Objetivo do projeto

Renderização 3D distribuída e Stereo Vision. Um líder eleito por Raft gera tarefas (slices de um torus paramétrico ou pares estéreo) e empurra para uma stream Redis. Os 6 nós consomem em paralelo via consumer group e devolvem pontos 3D que são desenhados em tempo real num frontend Three.js.

Sincronia de relógio offline

Força o horário do laptop em todos os nós

$NOW = Get-Date -Format "yyyy-MM-dd HH:mm:ss"
foreach ($n in 20..25) {
    Write-Host "Setando relógio em 192.168.1.$n -> $NOW" -ForegroundColor Yellow
    ssh [email protected].$n "sudo timedatectl set-ntp false && sudo date -s '$NOW'"
}
Por que isso é vital pro Raft? O algoritmo Raft usa timeouts de eleição (150–300ms). Se dois nós divergem mais do que isso, eles podem se considerar "líder" simultaneamente — split-brain. Em rede sem internet, sincronizar pelo laptop é o jeito mais simples de manter ±1s entre todos.

Usar Start-Job no PowerShell para instalar nos 6 nós ao mesmo tempo, transformando ~30min sequenciais em ~5min paralelos.
$IPS = 20..25
$jobs = @()

Write-Host "Instalando Docker em paralelo..." -ForegroundColor Cyan
foreach ($n in $IPS) {
    $ip = "192.168.1.$n"
    $jobs += Start-Job -ScriptBlock {
        param($alvo)
        ssh pi@$alvo @"
curl -fsSL https://get.docker.com | sh
sudo usermod -aG docker pi
sudo apt-get install -y docker-compose-plugin
sudo systemctl enable --now docker
"@
    } -ArgumentList $ip
}

Write-Host "Aguardando (~5 min)..." -ForegroundColor Yellow
Wait-Job -Job $jobs | Out-Null
Receive-Job -Job $jobs
Remove-Job -Job $jobs
Por que Start-Job e não ForEach-Object -Parallel? O `-Parallel` exige PowerShell 7+. Start-Job funciona em qualquer Windows com PS 5.1, não precisa de runspaces e o stdout do SSH é capturado certinho.

Deploy do Redis (com fix CRLF)
A pegadinha clássica do Windows: arquivos gerados pelo PowerShell vêm com CRLF + BOM, que faz o Redis explodir com erro fatal de parsing. A cura é gerar via [System.IO.File]::WriteAllText e substituir `\r\n` por `\n`.
!
Erro que isso evita: FATAL CONFIG FILE ERROR: Bad directive or wrong number of arguments na linha 1 do redis.conf. O culpado é o BOM invisível no início do arquivo.
$IPS = 20..25

# 1. redis.conf (sem replicas — performance máxima)
$RedisConf = @"
port 6379
cluster-enabled yes
cluster-config-file /data/redis/nodes.conf
cluster-node-timeout 2000
cluster-require-full-coverage no
cluster-migration-barrier 1
appendonly yes
appendfsync everysec
dir /data/redis
bind 0.0.0.0
protected-mode no
save 900 1
save 300 10
save 60 10000
"@.Replace("`r`n","`n")

# 2. docker-compose
$ComposeYml = @"
version: "3.9"
services:
  redis:
    image: redis:7-alpine
    container_name: cluster-redis
    restart: always
    network_mode: host
    command: redis-server /etc/redis/redis.conf
    volumes:
      - ./redis/redis.conf:/etc/redis/redis.conf:ro
      - /data/redis:/data/redis
    healthcheck:
      test: ["CMD","redis-cli","ping"]
      interval: 5s
      timeout: 3s
      retries: 5
"@.Replace("`r`n","`n")

New-Item -ItemType Directory -Force -Path ".\redis" | Out-Null
[System.IO.File]::WriteAllText("$PWD\redis\redis.conf", $RedisConf)
[System.IO.File]::WriteAllText("$PWD\docker-compose.redis.yml", $ComposeYml)

# 3. Distribuir e subir
foreach ($n in $IPS) {
    $ip = "192.168.1.$n"
    Write-Host "--- Deploy em $ip ---" -ForegroundColor Yellow
    ssh pi@$ip "sudo mkdir -p /data/redis && sudo chown -R pi:pi /data/redis && mkdir -p ~/cluster/redis"
    scp .\redis\redis.conf       pi@${ip}:~/cluster/redis/
    scp .\docker-compose.redis.yml pi@${ip}:~/cluster/
    ssh pi@$ip "cd ~/cluster && docker compose -f docker-compose.redis.yml up -d"
}
Por que `network_mode: host`? O Redis Cluster anuncia o IP de cada nó via gossip. Em modo bridge, o container vê seu IP interno (172.x) e os outros nós tentam se conectar nesse IP — falha. Em modo host, ele usa o IP real do RPi e tudo funciona.

Passo 7 · Infraestrutura
Inicializar o Redis Cluster (modo 6×0)
6 primários, 0 réplicas. A redundância sai da camada de aplicação via Redis Streams + recovery (passo 12), não da replicação do banco.
Por que 6×0 e não 3×1? Para cargas de renderização 3D, a métrica que importa é throughput de tarefas/segundo, não durabilidade absoluta de cada chave. Réplicas custariam metade do cluster sem nada em troca: se uma tarefa se perder, o recovery devolve para a fila em <10s. Trade-off consciente.
# 1. Criar o cluster
ssh [email protected] @"
docker exec cluster-redis redis-cli --cluster create \
  192.168.1.20:6379 192.168.1.21:6379 192.168.1.22:6379 \
  192.168.1.23:6379 192.168.1.24:6379 192.168.1.25:6379 \
  --cluster-replicas 0 --cluster-yes
"@

Start-Sleep -Seconds 5

# 2. Criar a stream e o consumer group
# OBS: o `{stream}` força tudo a cair no mesmo slot (hash tag)
ssh [email protected] 'docker exec cluster-redis redis-cli -c XGROUP CREATE "{stream}:tasks" "cluster-workers" $ MKSTREAM'

# 3. Status
ssh [email protected] "docker exec cluster-redis redis-cli cluster info | grep cluster_state"

ℹ
A hash tag {stream} nos nomes de chave (ex: {stream}:tasks, {stream}:results) força o Redis Cluster a colocá-las todas no mesmo slot. Isso permite usar comandos multi-key (XREADGROUP, XACK, XAUTOCLAIM) sem erro CROSSSLOT.

Passo 9 · Aplicação
Daemon Raft com pysyncobj
O cérebro do cluster. Roda em todos os 6 nós, conversa pela porta 4321 e elege um líder. Só o líder gera tarefas novas — os outros 5 ficam apenas como workers até o líder morrer e a eleição re-acontecer.
Por que pysyncobj? Implementa Raft puro em Python, sem depender de etcd/Consul. ~300 linhas de código C-extension, persistência em disco opcional, e a API é literalmente "herde de SyncObj e marque métodos com @replicated". Ideal pra não-Java.
A

O daemon

app/daemon.py
"""
Raft daemon: roda em cada nó, mantém estado replicado e elege um líder.
Só o líder chama gerar_tarefas(). Os outros viram workers passivos.
"""
import time, signal, sys, logging
from pysyncobj import SyncObj, SyncObjConf, replicated
from config import NODES, MY_IP, MY_NAME, RAFT_PORT

logging.basicConfig(
    level=logging.INFO,
    format="%(asctime)s [%(levelname)s] %(message)s",
)
log = logging.getLogger("daemon")


class ClusterState(SyncObj):
    """Estado replicado entre todos os nós via Raft."""
    def __init__(self):
        self_addr = f"{MY_IP}:{RAFT_PORT}"
        peers     = [f"{ip}:{RAFT_PORT}" for ip in NODES if ip != MY_IP]

        cfg = SyncObjConf(
            dynamicMembershipChange = True,
            appendEntriesUseBatch   = True,
            commandsWaitLeader      = True,
            connectionTimeout       = 3.5,
            raftMinTimeout          = 0.4,
            raftMaxTimeout          = 1.4,
        )
        super().__init__(self_addr, peers, cfg)

        self.__last_batch_id = 0
        self.__total_tasks   = 0

    @replicated
    def increment_batch(self) -> int:
        self.__last_batch_id += 1
        return self.__last_batch_id

    @replicated
    def add_tasks(self, n: int):
        self.__total_tasks += n

    def stats(self) -> dict:
        return {
            "last_batch_id": self.__last_batch_id,
            "total_tasks":   self.__total_tasks,
        }


def main():
    log.info(f"[{MY_NAME}] iniciando Raft em {MY_IP}:{RAFT_PORT}")
    state = ClusterState()

    # Esperar o cluster Raft formar quórum
    while state._getLeader() is None:
        log.info("aguardando eleição...")
        time.sleep(1)

    log.info(f"líder eleito: {state._getLeader()}")

    # Loop principal: só o líder produz tarefas
    from torus import generate_torus_batch
    import redis
    r = redis.Redis(host=MY_IP, port=6379, decode_responses=True)
    from config import STREAM_TASKS

    BATCH_SIZE = 200      # pontos por batch
    INTERVAL_S = 0.5      # gera 400 tarefas/s

    while True:
        try:
            if state._isLeader():
                batch_id = state.increment_batch(sync=True)
                points   = generate_torus_batch(batch_id, BATCH_SIZE)
                pipe = r.pipeline()
                for p in points:
                    pipe.xadd(STREAM_TASKS, p)
                pipe.execute()
                state.add_tasks(BATCH_SIZE, sync=False)
                log.info(f"[LÍDER] batch {batch_id} -> {BATCH_SIZE} tarefas")
            time.sleep(INTERVAL_S)
        except Exception as e:
            log.error(f"loop error: {e}")
            time.sleep(2)


def shutdown(signum, frame):
    log.info("encerrando...")
    sys.exit(0)


if __name__ == "__main__":
    signal.signal(signal.SIGTERM, shutdown)
    signal.signal(signal.SIGINT,  shutdown)
    main()
ℹ
Detalhes importantes:
• raftMinTimeout / raftMaxTimeout: 400–1400ms. Em LAN cabeada Gigabit isso é mais que suficiente e dá uma eleição em <2s.
• increment_batch(sync=True): bloqueia até o log Raft confirmar a maioria — garante batch_id único entre todos.
• add_tasks(sync=False): fire-and-forget, métricas não precisam de garantia forte.

Worker — consumir e processar tarefas
Cada nó (líder OU follower) roda um worker em paralelo ao daemon. Lê da stream via consumer group, processa o ponto 3D e devolve o resultado.
app/worker.py
"""
Worker: lê {stream}:tasks via consumer group, processa, escreve em {stream}:results.
Usa o nome do nó como consumer-id, então o recovery sabe quem morreu.
"""
import time, json, logging, signal, sys
import redis
from config import (
    MY_IP, MY_NAME, REDIS_PORT,
    STREAM_TASKS, STREAM_RESULTS, GROUP_NAME,
)
from torus import process_point

logging.basicConfig(level=logging.INFO,
    format="%(asctime)s [%(levelname)s] %(message)s")
log = logging.getLogger(f"worker-{MY_NAME}")


def connect():
    r = redis.Redis(host=MY_IP, port=REDIS_PORT, decode_responses=True)
    # Garante o group (idempotente — erro BUSYGROUP é esperado)
    try:
        r.xgroup_create(STREAM_TASKS, GROUP_NAME, id="$", mkstream=True)
    except redis.ResponseError as e:
        if "BUSYGROUP" not in str(e):
            raise
    return r


def main():
    r = connect()
    log.info(f"worker {MY_NAME} pronto, lendo de {STREAM_TASKS}")

    consumer = MY_NAME  # consumer-id estável

    while True:
        try:
            # Bloqueia até 2s esperando até 16 tarefas
            resp = r.xreadgroup(
                GROUP_NAME, consumer,
                {STREAM_TASKS: ">"},
                count=16, block=2000,
            )
            if not resp:
                continue

            _, messages = resp[0]
            ack_ids = []
            results = []

            for msg_id, fields in messages:
                t0 = time.perf_counter()
                point = process_point(fields)
                dt_ms = (time.perf_counter() - t0) * 1000

                results.append({
                    "task_id":   msg_id,
                    "worker":    MY_NAME,
                    "x":         point["x"],
                    "y":         point["y"],
                    "z":         point["z"],
                    "compute_ms": f"{dt_ms:.2f}",
                })
                ack_ids.append(msg_id)

            # Pipeline: gravar todos os resultados + ACK em uma RTT
            pipe = r.pipeline()
            for res in results:
                pipe.xadd(STREAM_RESULTS, res, maxlen=10_000, approximate=True)
            pipe.xack(STREAM_TASKS, GROUP_NAME, *ack_ids)
            pipe.execute()

            log.info(f"processadas {len(results)} tarefas")

        except redis.ConnectionError as e:
            log.error(f"redis caiu? {e} — reconectando em 2s")
            time.sleep(2)
            r = connect()
        except Exception as e:
            log.exception(f"erro inesperado: {e}")
            time.sleep(1)


def shutdown(*_):
    log.info("worker encerrado")
    sys.exit(0)

if __name__ == "__main__":
    signal.signal(signal.SIGTERM, shutdown)
    signal.signal(signal.SIGINT,  shutdown)
    main()
Por que MAXLEN aproximado? A stream de resultados cresce sem parar. maxlen=10_000, approximate=True mantém as últimas ~10k entradas (com algumas a mais por eficiência) — o frontend só precisa do que está visível na tela mesmo.
ℹ
O consumer = MY_NAME é a chave do recovery: quando o `rpi-03` morre, suas mensagens ficam em pending sob o consumer-id "rpi-03". O recovery (passo 12) vai detectar isso via XAUTOCLAIM e redistribuir.
Passo 11 · Aplicação
Cálculo do Torus 3D
A matemática é simples — um torus paramétrico com perturbação harmônica. O ponto da etapa não é a complexidade, é demonstrar que 6 nós dividem trabalho real e retornam pontos que desenham uma superfície coerente no frontend.
Equação paramétrica:
x = (R + r·cos(v + φ(t))) · cos(u)
y = (R + r·cos(v + φ(t))) · sin(u)
z = r · sin(v + φ(t))

Onde R é o raio maior, r o menor, u,v ∈ [0, 2π) e φ(t) uma função que faz a superfície "respirar" no tempo.
app/torus.py
"""
Geração e processamento de pontos sobre um torus paramétrico.
A geração é feita pelo líder (em batches); o processamento é o que
o worker executa por ponto.
"""
import math, time
from typing import List, Dict

R = 2.0   # raio maior
r = 0.7   # raio menor
RES_U = 80   # resolução em u
RES_V = 40   # resolução em v


def generate_torus_batch(batch_id: int, size: int) -> List[Dict[str, str]]:
    """
    O líder chama esta função. Cada item da lista vira um XADD.
    Os valores precisam ser strings (limitação do Redis Streams).
    """
    out = []
    t = time.time()
    for k in range(size):
        idx = (batch_id * size + k) % (RES_U * RES_V)
        i, j = divmod(idx, RES_V)
        u = (i / RES_U) * 2.0 * math.pi
        v = (j / RES_V) * 2.0 * math.pi
        out.append({
            "u": f"{u:.6f}",
            "v": f"{v:.6f}",
            "t": f"{t:.6f}",
            "batch": str(batch_id),
        })
    return out


def process_point(fields: Dict[str, str]) -> Dict[str, float]:
    """
    O worker chama isso. Carga útil real: trigonometria + perturbação.
    Em CPU ARM isso roda em ~30-80 µs por ponto. Aumentar a complexidade
    aqui é o que estressa o cluster (ex: ray-marching, voxel sampling).
    """
    u = float(fields["u"])
    v = float(fields["v"])
    t = float(fields["t"])
    age = time.time() - t

    # perturbação temporal — faz a superfície "respirar"
    phi = 0.5 * math.sin(2.0 * t * 0.3)
    rr  = r * (1.0 + 0.15 * math.sin(3 * u + t * 0.7))

    x = (R + rr * math.cos(v + phi)) * math.cos(u)
    y = (R + rr * math.cos(v + phi)) * math.sin(u)
    z = rr * math.sin(v + phi)

    return {"x": x, "y": y, "z": z, "age_ms": age * 1000}
★
Quer estressar de verdade? Substitua process_point por um ray-marching de SDF (signed distance field) com 64 passos por raio. Cada ponto vira ~5ms de cálculo, e o cluster passa a justificar sua existência. O esqueleto da fila não muda nada.

Recovery — devolver tarefas órfãs à fila
Sem réplicas no Redis, esse módulo é VITAL. Se um worker morre no meio do processamento, a tarefa fica "pending" eternamente. O recovery detecta isso e reentrega para outro worker.
!
Sem este módulo, num cluster 6×0 perder um nó significa perder definitivamente as tarefas que ele havia recebido e ainda não confirmado. Esta é a contrapartida que aceitamos no passo 7 quando escolhemos performance sobre replicação.
app/recovery.py
"""
Recovery loop: usa XAUTOCLAIM para tomar mensagens cujo consumer
não fez ACK em PENDING_TIMEOUT_MS. Roda APENAS no líder Raft —
assim evitamos múltiplos recoveries simultâneos competindo.
"""
import time, logging
import redis
from config import (
    MY_IP, MY_NAME, REDIS_PORT,
    STREAM_TASKS, GROUP_NAME, PENDING_TIMEOUT_MS, NODES,
)

logging.basicConfig(level=logging.INFO,
    format="%(asctime)s [%(levelname)s] %(message)s")
log = logging.getLogger("recovery")


def run(is_leader_fn, stop_fn=lambda: False):
    """
    is_leader_fn: callable que retorna True se este nó é o líder Raft.
    stop_fn:     callable que retorna True para encerrar o loop.
    """
    r = redis.Redis(host=MY_IP, port=REDIS_PORT, decode_responses=True)
    consumer_id = f"recovery-{MY_NAME}"
    cursor = "0-0"

    while not stop_fn():
        try:
            if not is_leader_fn():
                time.sleep(2)
                cursor = "0-0"
                continue

            # XAUTOCLAIM <key> <group> <consumer> <min-idle-ms> <start>
            cursor, claimed, _ = r.xautoclaim(
                STREAM_TASKS, GROUP_NAME, consumer_id,
                min_idle_time=PENDING_TIMEOUT_MS,
                start_id=cursor, count=64,
            )

            if claimed:
                log.warning(f"reclaimed {len(claimed)} tarefas órfãs")
                # Não processamos aqui — só reentregamos.
                # Como o `consumer_id` é o recovery, deixamos pendentes
                # com idle=0 para o próximo XREADGROUP de qualquer worker.
                # Truque: re-XADD as mesmas tarefas e ACK as antigas.
                pipe = r.pipeline()
                for msg_id, fields in claimed:
                    pipe.xadd(STREAM_TASKS, fields)
                    pipe.xack(STREAM_TASKS, GROUP_NAME, msg_id)
                pipe.execute()

            time.sleep(2)

        except Exception as e:
            log.error(f"loop error: {e}")
            time.sleep(3)


if __name__ == "__main__":
    # Modo standalone: assume sempre líder (para testes)
    run(is_leader_fn=lambda: True)
Por que re-XADD em vez de só XCLAIM? Se a gente apenas reclamasse para o consumer "recovery-rpi-01", a tarefa precisaria ser processada por esse processo. Re-publicando como uma nova mensagem, qualquer worker da consumer-group pode pegar — distribui melhor a carga.

API HTTP de status (FastAPI)
Cada nó expõe `/health`, `/queue/status` e `/results/recent`. O frontend Three.js consome `/results/recent` para desenhar; o watchdog consome `/health` para monitorar.
app/api.py
"""
FastAPI server: roda em todos os nós, porta 8000.
Endpoints:
  GET /health           — quem sou eu, sou líder, redis ok
  GET /queue/status     — tamanho da stream e pending count
  GET /results/recent   — últimos N pontos 3D (para o frontend)
"""
import time, redis
from fastapi import FastAPI
from fastapi.middleware.cors import CORSMiddleware
from config import (
    MY_IP, MY_NAME, REDIS_PORT,
    STREAM_TASKS, STREAM_RESULTS, GROUP_NAME, NODES,
)

app = FastAPI(title=f"cluster-rpi {MY_NAME}")
app.add_middleware(
    CORSMiddleware,
    allow_origins=["*"], allow_methods=["*"], allow_headers=["*"],
)

START_T = time.time()
r = redis.Redis(host=MY_IP, port=REDIS_PORT, decode_responses=True,
                socket_timeout=1.5)

# O daemon Raft precisa publicar quem é líder em algum lugar acessível
# pela API — usamos o próprio Redis como bus de status (chave volátil).
LEADER_KEY = "{stream}:cluster:leader"


@app.get("/health")
def health():
    try:
        r.ping()
        redis_ok = "ok"
    except Exception:
        redis_ok = "down"

    try:
        leader = r.get(LEADER_KEY)
    except Exception:
        leader = None

    return {
        "node":      MY_NAME,
        "ip":        MY_IP,
        "is_leader": leader == MY_IP,
        "leader":    leader,
        "redis":     redis_ok,
        "uptime_s":  int(time.time() - START_T),
        "raft_peers": len(NODES) - 1,
        "status":    "ok" if redis_ok == "ok" else "degraded",
    }


@app.get("/queue/status")
def queue_status():
    try:
        info = r.xinfo_stream(STREAM_TASKS)
        pending = r.xpending(STREAM_TASKS, GROUP_NAME)
        return {
            "stream_length": info["length"],
            "first_id":      info["first-entry"][0] if info["first-entry"] else None,
            "last_id":       info["last-entry"][0]  if info["last-entry"]  else None,
            "pending":       pending["pending"] if pending else 0,
        }
    except Exception as e:
        return {"error": str(e)}


@app.get("/results/recent")
def results_recent(limit: int = 500):
    """Retorna os últimos N pontos da stream de resultados."""
    try:
        # XREVRANGE pega do mais novo para o mais velho
        entries = r.xrevrange(STREAM_RESULTS, count=limit)
        return {
            "count": len(entries),
            "points": [
                {
                    "id":     msg_id,
                    "x":      float(f["x"]),
                    "y":      float(f["y"]),
                    "z":      float(f["z"]),
                    "worker": f["worker"],
                }
                for msg_id, f in entries
            ],
        }
    except Exception as e:
        return {"error": str(e), "points": []}

let nodeIdx = 0;
async function poll() {
  const node = NODES[nodeIdx++ % NODES.length];
  try {
    const res = await fetch(`http://${node}:8000/results/recent?limit=400`);
    const data = await res.json();
    if (!data.points) return;

    // Limpa buffers
    for (const k in groups) groups[k].buffer = [];

    for (const p of data.points) {
      const g = groups[p.worker];
      if (!g) continue;
      g.buffer.push(p.x, p.y, p.z);
    }

    let total = 0;
    for (const k in groups) {
      const arr = new Float32Array(groups[k].buffer);
      groups[k].geo.setAttribute('position', new THREE.BufferAttribute(arr,3));
      groups[k].geo.attributes.position.needsUpdate = true;
      total += arr.length / 3;

    // Líder
    const h = await fetch(`http://${node}:8000/health`).then(r=>r.json());
    document.getElementById('leader').textContent = h.leader || '?';
  } catch(e) { /* nó offline — round-robin tenta o próximo */ }

watchdog.py (no laptop)
"""
Watchdog do cluster RPi.
Uso: python3 watchdog.py
Requer: pip install requests
"""
import requests, time, os

NODES    = [f"192.168.1.{i}" for i in range(20, 26)]
INTERVAL = 2  # segundos

def clear(): os.system('cls' if os.name == 'nt' else 'clear')

def check(ip):
    try:
        h = requests.get(f"http://{ip}:8000/health", timeout=1.5).json()
        return {**h, "online": True}
    except Exception:
        return {"online": False, "node": "?", "ip": ip}

def queue(ip):
    try:
        return requests.get(f"http://{ip}:8000/queue/status", timeout=1.5).json()
    except Exception:
        return {}

print("\033[36mWatchdog ativo. Ctrl+C para sair.\033[0m")
try:
    while True:
        rows  = [check(ip) for ip in NODES]
        up    = [r for r in rows if r["online"]]
        down  = [r for r in rows if not r["online"]]
        leaders = [r for r in up if r.get("is_leader")]
        q = queue(up[0]["ip"]) if up else {}

        clear()
        print(f"┌{'─'*54}┐")
        print(f"│  Cluster RPi v2 — {time.strftime('%H:%M:%S')}   ({len(up)}/{len(NODES)} online){' '*8}│")
        print(f"├{'─'*54}┤")
        for r in rows:
            if r["online"]:
                star = "★" if r.get("is_leader") else " "
                color = "\033[32m" if r.get("status") == "ok" else "\033[33m"
                print(f"│ {color}✓\033[0m {r['ip']:<14} {star} {r['node']:<8} "
                      f"redis={r.get('redis','?'):<4} up={r.get('uptime_s',0):>5}s │")
            else:
                print(f"│ \033[31m✗\033[0m {r['ip']:<14}    OFFLINE{' '*22}│")
        print(f"├{'─'*54}┤")
        if q:
            print(f"│  fila: {q.get('stream_length',0):>6}   pendentes: {q.get('pending',0):>5}{' '*15}│")
        if leaders:
            print(f"│  líder: {leaders[0]['ip']}{' '*30}│")
        print(f"└{'─'*54}┘")

        if   len(leaders) == 0: print("\033[31m⚠ ALERTA: nenhum líder eleito!\033[0m")
        elif len(leaders) > 1:  print(f"\033[31m⚠ SPLIT-BRAIN: {len(leaders)} líderes!\033[0m")
        if down: print(f"\033[31m⚠ OFFLINE: {', '.join(d['ip'] for d in down)}\033[0m")

        time.sleep(INTERVAL)
except KeyboardInterrupt:
    print("\nencerrado.")
